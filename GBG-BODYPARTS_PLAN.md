# GBG-BODYPARTS — План за изграждане

> Проект: GK-bodyparts (едрогабаритни авточасти, body parts)
> Доставчик на данни: Georgakopoulos (код 36078)
> TecDoc Data Supplier Number: **3323**, марка GBG
> Целева платформа за дистрибуция: DVSE NEXT (Excel импорт по TecDoc стандарт)
> Изготвен: 2026-06-07

---

## 1. Цел и обхват

Две цели върху обща база.

1. **Захранване на DVSE NEXT** с четирите TecDoc Excel файла, генерирани автоматично от собствена нормализирана база.
2. **Собствен уеб каталог** за разглеждане и търсене по марка, модел и година, с цени, наличности и снимки.

И двете четат от една продукционна база `GBG-BODYPARTS`, отделена от дневния буфер.

### Корекция по NEXT (важно)

NEXT **не** връзва частите към автомобил сам. Връзката към конкретно превозно средство (KType) е наша работа и се извежда от текста на модела в отделна фаза. Затова OE номерата и KType връзката носят цялата тежест за приложимостта.

## 4. Нормализирана схема `GBG-BODYPARTS`

Идентичността е около `barcode` (basic item code). Всяка част получава собствен сурогатен `product_id`, който се мачва по `barcode` при всяко обновяване и **никога не се мени**. Това дава постоянен ключ за NEXT, ERP и уеб каталога.

### 4.1 `products` — една физическа част

| Колона         | Тип                                             | Източник / правило                      |
| -------------- | ----------------------------------------------- | --------------------------------------- |
| `product_id`   | INT IDENTITY PK                                 | Собствен сурогат, стабилен              |
| `barcode`      | NVARCHAR(20) UNIQUE                             | catalog.barcode (basic item code)       |
| `eng_descr`    | NVARCHAR(200)                                   | Представителен ред за този barcode      |
| `gr_descr`     | NVARCHAR(200)                                   | Представителен ред                      |
| `category_raw` | NVARCHAR(100)                                   | catalog.category (за мапинг към GenArt) |
| `genart_id`    | INT NULL                                        | От `map_category_genart`                |
| `weight`       | DECIMAL(8,3)                                    | catalog.weight                          |
| `cost_price`   | DECIMAL(10,2)                                   | catalog.price (доставна цена)           |
| `sale_price`   | DECIMAL(10,2) NULL                              | Собствена, по правило за надценка       |
| `side`         | NCHAR(2) NULL                                   | catalog.side (LE/RI)                    |
| `image_path`   | AS (`LEFT(barcode,4) + '/' + barcode + '.jpg'`) | Изводим, относителен път                |
| `stock_ath`    | BIT                                             | stock.ath                               |
| `stock_the`    | BIT                                             | stock.the                               |

## 5. Sync от буфер към продукция

Дневен скрипт след `import_gbg.py`, чрез `MERGE` (UPSERT), без TRUNCATE.

Логика за `products`:

1. Дедуплицирай `catalog` по `barcode` (представителен ред: например с най-висока цена или първи срещнат).
2. `MERGE` в `products` по `barcode`:
   - При съвпадение → обнови descr, цена, тегло, наличност, `last_seen`, `updated_at`. **Запази** `product_id`, `sale_price`, `is_active`.
   - При нов barcode → вмъкни, `first_seen = now`.
   - Липсващ в новия внос → маркирай `is_active = 0` (не трий, за история и ERP).
3. Презареди `applications`, `oem_numbers`, `cross_refs` по `product_id`.
4. Презапиши `genart_id` и `tecdoc_brand_id` от мапинг таблиците.

Изпълнение: дневно в 03:30 (след буферния внос в 03:00). Транзакция с rollback при грешка.

## 6. Мапинг таблици — изграждане

### 6.1 `map_brand_tecdoc` (58 марки)

Извличане на различните `name_raw` от catalog, после JOIN с таблицата на марките в `TECDOC_DATA` за `tecdoc_brand_id`. Ръчна проверка на несигурните. Еднократно, после само нови марки.

### 6.2 `map_category_genart` (~175 категории)

Същинската работа. Подход:

1. Извади 175-те категории, подредени по обем (брой артикули).
2. Почисти вариантите по марка качество (VALEO/NRF/NISSENS/MARELLI → един GenArt).
3. Махни доставчишкия шум (тип `1802 - GROUP FIAT KAT 20`).
4. За всяка категория намери GenArt ID от `TECDOC_DATA` (таблица генерични артикули), със стандартното описание на всички езици.
5. Започни от топ 30 категории, които покриват по-голямата част от артикулите.

> **Бонус:** щом GenArt е закачен, стандартното описание за NEXT, ERP и многоезичие идва наготово от `TECDOC_DATA` чрез JOIN по GenArt ID. Не пишем описания на ръка.

## 7. NEXT експорт — точно мапиране на полета

Ключ навсякъде: **Supplier Article Number = `<barcode> GBG`**. Един артикул на barcode. Supplier = `3323`.

### 7.1 `TM_Article_Details`

| Поле NEXT               | Източник                                                          |
| ----------------------- | ----------------------------------------------------------------- |
| Supplier                | `3323`                                                            |
| Supplier Article Number | `barcode + ' GBG'`                                                |
| Trader Article Number   | `barcode + ' GBG'`                                                |
| Additional Description  | `products.eng_descr`                                              |
| Pole Position           | `0`                                                               |
| Stock article           | `1` ако `stock_ath=1 OR stock_the=1`, иначе `0` (за потвърждение) |

### 7.2 `TM_Product_Groups`

| Поле NEXT               | Източник             |
| ----------------------- | -------------------- |
| Supplier                | `3323`               |
| Supplier Article Number | `barcode + ' GBG'`   |
| TecDoc GenArt ID        | `products.genart_id` |

> Артикули без мапнат GenArt се изключват от експорта, докато не получат GenArt. Това пази NEXT чист.

### 7.3 `TM_References` (няколко реда на артикул)

| Reference Type      | TecDoc ID | Източник                   | Reference Brand   |
| ------------------- | --------- | -------------------------- | ----------------- |
| OE number           | 4         | `oem_numbers.oem_code`     | `tecdoc_brand_id` |
| Utility number      | 2         | `barcode` (собствен номер) | празно            |

**Логика за Reference Brand при OE (критично):**

- Надеждният път: `catalog.genuine_code` идва с `catalog.brand` на същия ред. Тоест основният OE на дадена кола има ясна марка. Мапни `brand → tecdoc_brand_id`.
- Допълнителните OE от `oem_codes` нямат марка. При част с една марка → присвой нея. При част с няколко марки → нееднозначно, маркирай за ръчен преглед.

### 7.4 `TM_Attributes` (минимален старт)

| Поле NEXT               | Източник                                           |
| ----------------------- | -------------------------------------------------- |
| Supplier                | `3323`                                             |
| Supplier Article Number | `barcode + ' GBG'`                                 |
| TecDoc Criteria ID      | Зависи от GenArt                                   |
| Value                   | напр. `side` LE/RI → критерий за позиция на монтаж |
| Show in Article List    | `1`                                                |

Изворните данни носят почти само `side` и `weight`. Пълните TecDoc критерии са стотици на GenArt. **Старт с минимум** (позиция на монтаж от `side`), разширяване после. Не блокира пускането.

## 8. Връзка към автомобил (KType) — отделна фаза

NEXT няма да върже частите към коли. Изворът дава свободен текст тип `MERCEDES E CLASS 2002-2009 (W211)`.

Подход за извеждане на KType от текста:

1. Раздели текста на марка, модел, диапазон години и шаси код (`W211`).
2. JOIN срещу таблиците за превозни средства в `TECDOC_DATA` по марка плюс модел плюс години.
3. Шаси кодът (`W211`) е силен сигнал, TecDoc често носи типовото означение.
4. Резултатът е набор от KType кандидати. Фъзи мач, изисква ръчен преглед на несигурните.

**Отворен въпрос:** четирите NEXT шаблона не включват файл за връзка артикул към KType. Трябва да потвърдим дали NEXT приема пети файл за vehicle links, или връзката се качва другаде. Това решава как доставяме KType на платформата.

## 9. Собствен уеб каталог

Чете от `GBG-BODYPARTS`. Функции:

- Търсене по марка, модел и година (използва `applications` и по-късно KType).
- Търсене по OE номер и по cross-ref.
- Страница на артикул: описание (стандартно от GenArt), категория, тегло, наличност по склад (зелено/червено, без бройки, по политика на доставчика), снимка, цена (`sale_price`).
- Филтри по категория и страна (LE/RI).

**Решено:**

- **Снимки.** NAS, монтиран на сървъра в `/mnt/gbg-data/Parts/`. Именувани по barcode, в папка от първите четири цифри (model_code). Пример `/Parts/0188/018806270.jpg`. Пътят е изводим само от barcode: `LEFT(barcode,4) + '/' + barcode + '.jpg'`, без отделна таблица за мапинг. Същите снимки вървят и към NEXT, понеже TecDoc пакетът приема изображения по артикул. Проверка за наличие на файл при сервиране (не всеки barcode има снимка).
- **Продажна цена.** Идва от ERP. Не я мислим сега. Полето `sale_price` остава, ERP го пълни по-късно.
- **Технология.** Ориентир Next.js, клиент-сървър подход. Решава се на фаза 8.
